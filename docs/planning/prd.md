## 1. 프로젝트 개요

### 1.1 배경 및 문제 정의

팀 기반 멀티플레이 FPS 장르는 전 세계적으로 높은 플레이어 수와 시장 규모를 유지하고 있다. 기존 3D FPS 게임과 달리, 2D 탑다운 방식은 낮은 하드웨어 요구사항과 직관적인 전술 시각화를 제공하여 캐주얼 및 코어 플레이어 모두에게 접근성이 높다.

현재 시장에는 고사양 PC를 요구하는 3D 팀 FPS가 주류를 이루고 있어, 낮은 사양의 환경에서 브라우저 또는 경량 클라이언트로 팀 전술 전투를 즐길 수 있는 게임이 부족하다.

### 1.2 제품 비전

> 누구나 브라우저에서 바로 접속하여, 팀워크와 전술로 승부를 겨루는 2D 탑다운 멀티플레이 FPS 게임을 제공한다.

### 1.3 목적

- 팀 기반 전술 전투 경험을 제공하는 2D 멀티플레이 FPS 게임 출시
- 캐주얼 플레이어와 코어 플레이어 모두가 즐길 수 있는 접근성 확보
- MVP 출시 후 플레이어 피드백을 통한 빠른 반복 개선

### 1.4 성공 지표 (KPI)

| 지표 | 목표값 | 측정 방법 |
|------|--------|-----------|
| 동시 접속자 네트워크 레이턴시 | 100ms 이하 | Unity Stats 측정 |
| 재플레이율 | 첫 플레이 후 60% 이상 | 내부 테스트 그룹 측정 |
| 평균 세션 시간 | 15분 이상 | 클라이언트 세션 로그 |
| MVP 4명 동시 접속 안정성 | 오류 없이 1매치 완주 | 내부 자동화 테스트 |
| 라운드 시스템 승패 판정 오류 | 0건 | 자동화 테스트 |

## 2. 목표

### 2.1 비즈니스 목표

- MVP를 M2(Week 9, 2026-06-06)까지 완성하여 핵심 가치를 조기 검증한다.
- 정식 출시(M5, 2026-08-23)를 통해 초기 유저 기반을 확보한다.
- WebGL 빌드로 설치 없이 브라우저 접속을 지원하여 진입 장벽을 낮춘다.

### 2.2 사용자 목표

- 복잡한 설치나 사양 제한 없이 팀 전술 FPS 게임을 즐길 수 있다.
- 친구와 함께 또는 자동 매칭으로 공정한 4v4 대전에 참가할 수 있다.
- 게임 결과와 개인 기여 지표(킬/데스/어시스트)를 확인하며 실력 향상을 인지할 수 있다.

### 2.3 기술 목표

- Unity 2022 LTS + Netcode for GameObjects(NGO) 기반으로 서버 권위 멀티플레이 구조를 구현한다.
- MVP 단계에서는 Unity Gaming Services(UGS) Relay를 사용하고, 이후 자체 서버로 전환 계획을 수립한다.
- WebGL 빌드를 포함하여 브라우저에서 즉시 테스트 가능한 환경을 제공한다.
- 단위 테스트 커버리지 80% 이상, 코드 리뷰 90점 이상을 유지한다.

## 3. 이해관계자 및 페르소나

### 3.1 주요 사용자 페르소나

#### 페르소나 A — 코어 게이머 "전략형 팀플레이어"

| 항목 | 내용 |
|------|------|
| 대표 인물 | 박준혁 (22세, 대학생) |
| 플레이 환경 | PC, 하루 2~4시간 |
| 주 플레이 장르 | 팀 FPS (CS2, Valorant), 탑다운 슈터 |

**핵심 Pain Point**
- 솔로 플레이 시 전략성이 무너지는 게임에 실망한다.
- 스킬 격차가 커서 신규-기존 유저 혼합 매칭 품질이 낮다.
- 접속 대기 시간이 30초 이상이면 이탈한다.

**기대 결과**
- 팀워크를 발휘한 전술 전투로 승리하는 경험
- 킬/데스 비율 및 팀 기여도 지표 확인

#### 페르소나 B — 캐주얼 게이머 "가볍게 즐기는 플레이어"

| 항목 | 내용 |
|------|------|
| 대표 인물 | 이수진 (19세, 고등학생) |
| 플레이 환경 | PC 또는 노트북, 하루 1시간 이내 |
| 주 플레이 장르 | 아케이드, 모바일 슈터 |

**핵심 Pain Point**
- 조작이 복잡하거나 학습 곡선이 가파르면 흥미를 잃는다.
- 게임 설치 과정이 번거롭거나 사양이 높으면 포기한다.
- 팀원이 극단적으로 경쟁적이면 불쾌한 경험을 한다.

**기대 결과**
- 브라우저에서 바로 접속하여 친구와 즐거운 플레이
- 짧은 세션(15분 내외)에서도 성취감을 느낄 수 있는 라운드 구조

#### 페르소나 C — 인디 게임 콘텐츠 크리에이터

| 항목 | 내용 |
|------|------|
| 대표 인물 | 김민준 (27세, 유튜버/스트리머) |
| 플레이 환경 | 고사양 PC, 방송 장비 보유 |
| 주 플레이 장르 | 인디 게임, 멀티플레이 게임 전반 |

**핵심 Pain Point**
- 방송 중 프레임 드롭·크래시 발생 시 방송 중단 위험이 있다.
- 게임 반복성이 높으면 시청자 이탈이 발생한다.

**기대 결과**
- 60FPS 안정적 유지로 방송 품질 확보
- 인상적인 클립(킬, 역전 라운드 등)이 빈번하게 발생하는 게임플레이

### 3.2 이해관계자

| 이해관계자 | 역할 | 주요 관심사 |
|-----------|------|------------|
| 최성민 (게임 개발팀 리드 / 비전) | 오케스트레이터 | 일정 준수, 품질, 팀 생산성 |
| 정다은 (QA 엔지니어) | 품질 보증 | 동기화 버그 사전 발견, 테스트 자동화 |
| 한승우 (백엔드 개발자) | 서버·네트워크 개발 | 서버 안정성, 매칭 시스템, 실시간 동기화 |

### 3.3 핵심 사용자 여정

```
[발견] 유튜브/SNS 영상 시청 또는 친구 추천
    ↓
[접속] 브라우저(WebGL) 또는 클라이언트 실행 (설치 최소화)
    ↓
[온보딩] 기본 조작 학습 (직관적 WASD + 마우스)
    ↓
[매칭] 로비에서 방 생성·참가 또는 자동 매칭
    ↓
[플레이] 4v4 전술 전투 (라운드 기반 Best of 5)
    ↓
[결과] 스코어보드 확인, 개인/팀 기여도 피드백
    ↓
[재플레이] 다음 매치 참여 또는 로비 복귀
```

## 4. 기능 요구사항

> 우선순위 레이블: **Must** = MVP 필수 / **Should** = M4 목표 / **Could** = Post-MVP 검토

### 4.1 게임플레이

#### FR-001 캐릭터 이동 [Must]

- WASD 키로 8방향 이동 (대각선 포함, Vector2.normalized 적용)
- 이동 속도: 기본 5 units/sec (무기에 따라 최대 20% 감소 가능)
- 마우스 커서 방향으로 캐릭터 회전 (atan2 계산)
- 장애물 및 맵 경계 충돌 시 이동 불가
- 사망 상태(isDead = true)에서 이동 입력 무시

#### FR-002 무기 시스템 [Must / Should]

- **MVP (Must)**: 소총(Rifle) 1종
- **M4 (Should)**: 산탄총(Shotgun), 저격총(Sniper) 추가 (총 3종)
- 숫자키 1-3 또는 마우스 휠로 무기 교체 (M4)
- R키 수동 재장전, 탄창 소진 시 자동 재장전 시작

| 무기 | 데미지 | RPM | 탄창 | 재장전(초) | 사거리 | 특성 |
|------|--------|-----|------|-----------|--------|------|
| 소총 | 25 | 600 | 30 | 2.0 | 무제한 | 단일 직선 탄 |
| 산탄총 | 15x6 | 80 | 6 | 2.5 | 중거리(10u) | 부채꼴 6발, ±15도 산개 |
| 저격총 | 100 | 40 | 5 | 3.0 | 무제한 | 관통, 우클릭 줌인 |

#### FR-003 HP 시스템 [Must]

- 기본 HP: 100
- 탄환 충돌 시 데미지 계산 적용 (IDamageable 인터페이스)
- 헤드샷 영역: 데미지 1.5배 / 몸통: 1.0배
- HP 0 이하 시 사망, 라운드 시작 시 100으로 전체 회복
- 서버 권위 처리: NetworkVariable 기반 HP 동기화

#### FR-004 라운드 시스템 [Must]

- Best of 5: 먼저 3라운드 획득 팀이 매치 승리
- 라운드 종료 조건: 한 팀 전원 사망
- 라운드 시작 카운트다운: 5초 (이동·사격 잠금)
- 라운드 종료 후 결과 팝업: 3초 표시
- 두 팀 동시 전멸 시: 라운드 무효 처리 후 재진행
- 라운드 최대 시간 초과 시: 생존 인원 더 많은 팀 승리, 동수 시 무효

#### FR-005 팀 시스템 [Must]

- Red Team / Blue Team 2팀, 팀당 최대 4명
- 팀 색상으로 아군·적군 시각적 구분
- 아군 피해 없음 (팀킬 불가)
- 팀별 스폰 구역 지정 (맵 양 끝)
- **팀 배정 방식**: 방 입장 시 서버가 Red/Blue 인원 균형(차이 1 이하)을 기준으로 랜덤 자동 배정
- **팀 변경**: 입장 후 빈 슬롯(상대 팀 인원이 자신 팀보다 적은 경우)이 있을 때에 한해 팀 변경 버튼 활성화; 팀 변경 시 준비 완료 상태 자동 해제

### 4.2 네트워크 및 멀티플레이

#### FR-006 방(Room) 시스템 [Must]

- 방 생성: 방 이름(최대 20자), 최대 인원(4/6/8) 설정
- 방 참가: 방 코드 또는 목록에서 선택
- 최대 8명(4v4) 수용
- 호스트 연결 끊김 시 가장 낮은 클라이언트 ID로 자동 호스트 이전

#### FR-007 매칭 시스템 [Must]

MVP에 아래 두 가지 방식 모두 포함한다.

| 방식 | 설명 | 팀 배정 |
|------|------|---------|
| 빠른 매칭 | 버튼 하나로 자동 방 배정 | 팀 랜덤 자동 배정 |
| 방 탐색 | 방 목록(로비 브라우저)에서 직접 방 선택 후 입장 | 팀 랜덤 자동 배정 |

**빠른 매칭 세부 요구사항**
- 메인 로비 화면에서 "빠른 매칭" 버튼 1회 클릭으로 즉시 자동 방 배정
- 대기 가능한 방(인원 미달, 게임 시작 전)에 자동 입장; 없을 경우 신규 방 자동 생성
- 매칭 대기 시간 목표: 30초 이내
- 매칭 대기 30초 초과 시 "대기 중..." 상태 안내 메시지 화면 표시
- 방 배정 후 팀 랜덤 자동 배정 (Red/Blue 인원 균형 우선)

**방 탐색 세부 요구사항**
- 로비 브라우저: 현재 참가 가능한 방 목록 표시 (방 이름, 현재 인원/최대 인원, 상태)
- 방 목록 갱신 주기: 5초
- 원하는 방 선택 후 "입장" 버튼으로 직접 참가
- 입장 후 팀 랜덤 자동 배정 (Red/Blue 인원 균형 우선)
- 방이 가득 찼거나 이미 게임 시작된 경우 입장 불가 처리 및 안내 메시지 표시

**공통 팀 배정 규칙**
- 팀 배정 기준: Red/Blue 팀 인원 차이 1 이하 유지
- 팀 배정은 서버가 결정 (클라이언트 조작 불가)
- 레벨·랭크 기반 매칭: Post-MVP (M3 이후) 검토

#### FR-008 게임 상태 동기화 [Must]

- 동기화 대상 및 방법:

| 데이터 | 동기화 방법 | 업데이트 주기 |
|--------|------------|--------------|
| 플레이어 위치·회전 | NetworkTransform (보간 적용) | 매 프레임 |
| HP | NetworkVariable\<float\> | 변경 시 즉시 |
| 사망 상태 | NetworkVariable\<bool\> | 변경 시 즉시 |
| 탄환 발사 이벤트 | ServerRpc → ClientRpc | 이벤트 발생 시 |
| 라운드 상태 | NetworkVariable\<RoundState\> | 변경 시 즉시 |

- 클라이언트 예측(Client Prediction) + 서버 보정(Reconciliation) 적용
  - 오차 0.1 units 이하: 보간으로 자연스러운 보정
  - 오차 0.1 units 초과: 즉시 서버 위치로 스냅

#### FR-009 Relay 서버 [Must]

- **확정**: MVP 단계에서는 Unity Gaming Services(UGS) Relay 사용
- **확정**: WebGL 빌드를 포함하므로 UGS Relay는 필수 조건
- Post-MVP: 자체 Relay 서버 구축으로 전환 계획 수립 (M5 이후)

### 4.3 맵 및 환경

#### FR-010 맵 요구사항 [Must / Should]

- **MVP (Must)**: 기본 맵 1종 (실내형, 박스 레이아웃)
- **M4 (Should)**: 맵 2종 (실내형 + 실외형/오픈형)
- 맵 크기: 화면 기준 20x20 단위 이상
- 장애물: 최소 10개 이상 배치 (엄폐물 역할)
- 맵 경계: 캐릭터 이탈 불가 처리 (경계 콜라이더)
- 팀별 스폰 구역: 맵 양 끝에 분리 배치

### 4.4 UI / UX

#### FR-011 HUD [Must]

- HP 바: 현재/최대 HP 색상 표시 (녹색 70%↑ / 노란색 30~70% / 빨간색 점멸 30%↓)
- 탄약 표시: 현재 탄창 / 보유 탄약
- 팀 스코어: Red Team vs Blue Team 라운드 수
- **킬 피드 [MVP Must]**: 우측 상단, 최근 3건, 4초 후 페이드 아웃 (형식: 킬러 -- 무기 --> 사망자)
  - MVP 필수 포함으로 확정; MVP 제외 범위에서 제거됨

#### FR-012 로비 UI [Must]

- 방 생성 / 방 참가 버튼
- 방 목록 화면 (방 이름, 현재 인원/최대 인원, 상태)
- 팀 선택 화면 (Red / Blue, 팀 인원 차이 1 이하 조건)
- 준비 완료 버튼 (전원 준비 완료 시 호스트 게임 시작 활성화)

#### FR-013 결과 화면 [Must]

- 승리/패배 명확한 표시
- 팀별 총 라운드 수

## 6. 기술 스택 및 아키텍처 개요

| Relay 서버 (MVP) | Unity Gaming Services (UGS) Relay | WebGL 필수 조건 |

| 언어 | C# | |
| 빌드 타겟 | Windows PC, WebGL | macOS/Linux는 M5 이후 검토 |
| 버전 관리 | Git / GitHub | Git Flow 브랜치 전략 |
| CI/CD | GitHub Actions | |
| 테스트 | Unity Test Runner (NUnit) | |

### 6.2 아키텍처 개요

```
[클라이언트 - Windows/WebGL]
    ├── Unity 2D 씬 (탑다운 뷰)
    ├── PlayerMovement, WeaponController, HealthSystem
    ├── HUDController, LobbyUIController, ResultUIController
    └── NGO 클라이언트 레이어 (NetworkTransform, NetworkVariable)
            │
            │ UGS Relay (MVP) / 자체 서버 (Post-MVP)
            │
[서버 (Host 클라이언트 또는 전용 서버)]
    ├── 서버 권위 처리 (이동 검증, 데미지 계산)
    ├── RoundManager, TeamManager, GameManager
    └── NGO 서버 레이어 (ServerRpc 수신, ClientRpc 발신)
```

```

        TeamManager.cs          — 팀 구성, 전멸 감지
```

## 11. 용어 정의 (Glossary)

| 용어 | 정의 |
|------|------|
| 라운드 | 한 팀이 전멸할 때까지의 단위 전투 |
| 매치 | 여러 라운드로 구성된 하나의 게임 세션 (Best of 5) |
| 스폰 | 라운드 시작 시 캐릭터가 배치되는 지점 |
| 서버 권위 | 게임 상태의 최종 판단을 서버가 담당하는 방식 (Anti-cheat 기반) |
| 킬 피드 | 화면에 표시되는 실시간 킬 이벤트 목록 |
| HUD | Heads-Up Display, 게임 중 화면에 표시되는 정보 패널 |
| NGO | Unity Netcode for GameObjects, Unity 공식 멀티플레이 프레임워크 |
| UGS | Unity Gaming Services, Unity 공식 클라우드 서비스 (Relay 포함) |
| Relay 서버 | 클라이언트 간 직접 연결 대신 중계 역할을 하는 서버 (NAT 우회) |
| WebGL | 브라우저에서 실행 가능한 Unity 빌드 타겟 |
| Best of 5 | 5라운드 중 먼저 3라운드 획득 팀이 승리하는 방식 |
| 서버 보정 | 클라이언트 예측과 서버 처리 결과의 오차를 서버 기준으로 수정하는 과정 |
| MVP | Minimum Viable Product, 핵심 가치 검증에 필요한 최소 기능 제품 |
| PoC | Proof of Concept, 기술 실현 가능성을 사전 검증하는 작업 |
