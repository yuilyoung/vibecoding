## ADR-001: Unity Netcode for GameObjects (NGO) 선택

### 상태: 승인됨

### 컨텍스트

4v4 실시간 멀티플레이 게임을 구현하기 위해 네트워크 프레임워크를 선정해야 한다.
Unity 생태계 내에서 주요 선택지는 NGO, Mirror, Photon PUN2, Fish-Net이 있다.
WebGL 빌드 지원, UGS Relay 연동, Unity 2022 LTS 공식 지원이 핵심 조건이다.

### 결정

Unity Netcode for GameObjects (NGO)를 선택한다.

### 이유

- Unity 2022 LTS와 공식 통합: Unity Editor에서 Package Manager로 직접 설치 및 지원
- UGS Relay / Lobby / Authentication과 네이티브 연동: 별도 어댑터 없이 SDK 통합
- WebGL 빌드 공식 지원: WebGL 환경에서의 멀티플레이가 필수 요구사항
- 서버 권위 구조 내장: NetworkVariable, ServerRpc, ClientRpc, NetworkTransform이 서버 권위 패턴에 최적화
- Host-Client 모드 지원: MVP 단계 전용 서버 없이 호스트 클라이언트로 운영 가능

### 트레이드오프

- 장점: Unity 공식 지원, UGS 생태계 완전 통합, WebGL 지원, 무료
- 단점: Mirror 대비 커뮤니티/레퍼런스 적음, NGO 버전 간 API 변경 이력 존재, 고급 물리 동기화 기능 부족

### 대안 검토

- **Mirror**: 오픈소스, 커뮤니티 풍부하나 UGS Relay 네이티브 연동 미지원, WebGL 지원 불안정
- **Photon PUN2**: 빠른 구축 가능하나 무료 CCU 한계(20명), 유료 비용 발생, 서버 권위 구현 복잡
- **Fish-Net**: 성능 우수하나 UGS 통합 없음, 팀 내 학습 비용 높음

## ADR-002: UGS Relay + Lobby 사용 (MVP 단계)

### 상태: 승인됨

### 컨텍스트

WebGL 빌드는 직접 소켓 서버 연결이 불가능하며 NAT 우회(Relay)가 필수다.
MVP 단계에서 자체 서버 인프라 구축에 투자하는 것은 일정 위험이 높다.

### 결정

MVP 단계에서 Unity Gaming Services(UGS) Relay 및 Lobby API를 사용한다.
Post-MVP(M5 이후)에서 자체 Relay 서버 전환을 별도 ADR로 검토한다.

### 이유

- WebGL 환경에서 NAT 우회를 위한 Relay 서버가 필수이며, UGS Relay는 무료 플랜 제공
- UGS Lobby API로 방 생성·목록·참가 기능을 빠르게 구현 가능
- NGO와 네이티브 통합으로 추가 개발 비용 없음
- MVP 완료 전 인프라 구축에 리소스 소모를 방지

### 트레이드오프

- 장점: 빠른 MVP 구현, 인프라 비용 없음(무료 플랜 범위), WebGL 완전 지원
- 단점: UGS 서비스 장애 시 의존성 리스크, 무료 플랜 CCU 한계, 장기 비용 발생 가능

### 대안 검토

- **자체 Relay 서버 (MVP 단계 구축)**: 인프라 구축 기간(2~3주) 일정 초과 리스크 → 기각
- **LAN 전용 프로토타입**: WebGL 지원 불가, 요구사항 미충족 → 폴백으로만 유지

### 후속 계획

- M5 이후: 자체 Relay 서버(예: coturn) 또는 게임 서버(예: Agones on GKE) 전환 ADR 별도 작성

## ADR-003: Windows + WebGL 듀얼 빌드 타겟

### 상태: 승인됨

### 컨텍스트

제품 비전은 "브라우저에서 바로 접속하여 즐기는 게임"이다.
동시에 Windows 네이티브 클라이언트로 더 나은 성능과 조작감을 제공해야 한다.

### 결정

Windows PC 빌드와 WebGL 빌드를 모두 MVP에 포함한다.

### 이유

- WebGL: 설치 없이 브라우저 접속으로 진입 장벽 최소화 (캐주얼 페르소나 B 대응)
- Windows: 60FPS 안정 성능, 크리에이터 방송 환경 대응 (페르소나 C 대응)
- 두 빌드 모두 UGS Relay를 통해 같은 서버에 접속하므로 크로스플레이 가능

### 트레이드오프

- 장점: 최대 접근성, 크로스플레이 지원, 브라우저 테스트 용이
- 단점: WebGL 빌드 최적화 추가 작업 필요 (Brotli 압축, 메모리 제약), CI 빌드 시간 증가

### 대안 검토

- **Windows 전용**: 접근성 요구사항 미충족 → 기각
- **WebGL 전용**: 60FPS 안정성 및 방송 환경 지원 불가 → 기각
- **macOS 추가**: 우선순위 낮음, M5 이후 검토

## ADR-004: Host-Client 구조 (전용 서버 없음, MVP)

### 상태: 승인됨

### 컨텍스트

MVP 단계에서 게임 서버를 별도 운영하면 인프라 비용과 구축 시간이 증가한다.
NGO는 Host-Client 모드를 지원하며, 호스트가 서버 로직을 실행할 수 있다.

### 결정

MVP에서는 전용 게임 서버 없이, 방장 클라이언트가 NGO Host 역할을 겸한다.
호스트 이탈 시 NGO Host Migration으로 다음 클라이언트에게 호스트 권한을 이전한다.

### 이유

- 서버 인프라 구축 비용 및 시간 절감
- NGO Host-Client 모드는 Unity 공식 지원 패턴
- MVP 기간(8주) 내 인프라 운영 리스크 제거

### 트레이드오프

- 장점: 인프라 비용 없음, 빠른 구현, NGO 네이티브 지원
- 단점: 호스트 연결 품질에 의존, 호스트 유리(Host Advantage) 가능성, Host Migration 복잡도

### 대안 검토

- **전용 Dedicated Server**: 서버 권위 완전 분리로 공정성 높으나 MVP 일정 초과 → Post-MVP 전환
- **P2P Mesh**: NAT 우회 어렵고 서버 권위 구현 불가 → 기각

### 후속 계획

- M5 이후: Dedicated Server 구조 전환 ADR 작성 (Unity GameServer Hosting 또는 자체 호스팅)

## ADR-005: 킬 피드 MVP Must 확정

### 상태: 승인됨 (2026-04-05 변경 반영)

### 컨텍스트

초기 PRD 초안에서 킬 피드(Kill Feed)는 M4(Should) 범위로 설정되어 있었다.
그러나 4v4 FPS에서 킬 피드는 팀 상황 파악과 게임 피드백의 핵심 요소이며,
MVP 플레이어 경험에 직결된다.

### 결정

킬 피드를 MVP Must로 확정한다. (PRD v1.1 반영)

### 이유

- 킬 피드 없이는 팀 전멸 상황 파악이 HUD만으로 부족
- 킬 피드는 FR-011 HUD의 일부로, 별도 복잡한 시스템이 아님 (UIManager + OnKillEvent)
- 구현 공수: UI 컴포넌트 + 이벤트 수신으로 1~2일 수준
- FSD에 이미 킬 피드 설계(형식, 표시 건수, 페이드 시간)가 포함되어 있음

### 트레이드오프

- 장점: MVP 플레이어 경험 완성도 향상, 추가 구현 비용 최소
- 단점: M2 작업 범위 소폭 증가

## ADR-006: 팀 배정 방식 — 서버 랜덤 자동 배정 + 조건부 팀 변경

### 상태: 승인됨 (2026-04-05 변경 반영)

### 컨텍스트

방 입장 시 팀 배정 방식과 팀 변경 허용 조건을 명확히 정의해야 한다.
클라이언트가 자유롭게 팀을 선택하면 한 팀으로 쏠림이 발생할 수 있다.

### 결정

- 방 입장 시: 서버가 Red/Blue 인원 균형(차이 1 이하)을 기준으로 랜덤 자동 배정
- 팀 변경: 상대 팀에 빈 슬롯이 있을 때에 한해 팀 변경 버튼 활성화
- 팀 변경 시: 준비 완료 상태 자동 해제

### 이유

- 팀 불균형 방지로 공정한 4v4 구성 보장
- 서버가 배정을 결정하여 클라이언트 조작 불가
- 빈 슬롯 조건으로 팀 변경 후에도 균형 유지
- 준비 완료 자동 해제로 팀 변경 후 의도치 않은 게임 시작 방지

### 트레이드오프

- 장점: 팀 균형 보장, 보안 (서버 결정), 단순 UI
- 단점: 플레이어 팀 선호도 자유 선택 불가 (친구와 같은 팀 원하는 경우 제한)

### 대안 검토

- **자유 팀 선택**: 팀 쏠림 문제 발생, FPS 게임 공정성 훼손 → 기각
- **팀 고정 (변경 불가)**: 입장 직후 연결 불균형 발생 시 수정 수단 없음 → 기각

## ADR-007: 2D 탑다운 뷰 선택

### 상태: 승인됨

### 컨텍스트

2D FPS 게임의 시점 방식으로 탑다운(Top-Down)과 사이드뷰(Side-View/Platformer)를 검토했다.

### 결정

2D 탑다운(Top-Down) 뷰를 채택한다.

### 이유

- 탑다운 FPS는 전술적 시야와 팀 포지션 파악이 용이 (CS2, Hotline Miami 레퍼런스)
- WASD 이동 + 마우스 조준 조합이 자연스럽게 매핑
- 팀 스폰 구역 양 끝 배치가 직관적
- 사이드뷰는 점프/중력 시스템 추가 복잡도 발생

### 트레이드오프

- 장점: 직관적 전술 시야, 조작 단순, 맵 설계 자유도
- 단점: 탄환 표현에서 Z축 없음 (높이 차이 표현 불가), 일부 시각적 단조로움

## ADR-008: 오브젝트 풀링 (탄환)

### 상태: 승인됨

### 컨텍스트

4v4 전투에서 소총(RPM 600) 기준 초당 최대 80발 탄환이 생성/소멸된다.
매 발사마다 Instantiate/Destroy를 반복하면 GC 압박 및 FPS 저하가 발생한다.

### 결정

BulletController에 오브젝트 풀(Object Pool)을 적용한다.
Unity 2021 이후 내장된 `ObjectPool<T>` 또는 커스텀 풀을 사용한다.

### 이유

- GC 할당 제거로 FPS 안정성(60+ 유지) 보장
- 탄환 Prefab 재사용으로 메모리 효율화
- NFR-001(60 FPS), NFR-003(8명 동시 FPS 유지) 충족을 위한 필수 최적화

### 트레이드오프

- 장점: GC 최소화, FPS 안정, 초기 풀 크기 조정으로 메모리 제어
- 단점: 초기 풀 크기 과도 설정 시 메모리 낭비, 풀 부족 시 확장 로직 필요
